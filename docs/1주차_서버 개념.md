## 서버개념
### 서버와 클라언트란?
#### **서버**: 클라이언트의 요청을 받아 처리하는 주체. html 페이지를 넘겨주거나, 데이터를 조회하여 JSON 형태로 전달하는 등 <br> **클라이언트**: 서버에게 웹 페이지, 이미지 등의 리소스 요청을 보내는 주체.
### J2EE(JavaEE), EJB, Serverlet이란?(자바 웹 프레임워크의 역사)
#### **J2EE**: Java2 Enterprise Edition. 자바를 이용하여 엔터프라이즈 환경(대규모의 동시 접속과 유지가 가능한 다양한 시스템의 연동 네트워크 기반 총칭)의 애플리케이션을 개발하는 데 필요한 라이브러리들을 모아놓은 플랫폼. 구성 요소로 Serverlet, JSP, EJB 등이 있음.
#### EJB: Enterprise Java Bean. 엔터프라이즈 환경의 시스템을 구현하기 위한 서버 측 컴포넌트 모델.
##### Java Bean: 데이터 표현을 위해 만들어진 재사용 가능한 자바 클래스. 아래의 규격에 따라 만들어짐.
- 클래스는 패키지화 해야 한다.
- 멤버 변수는 프로퍼티라 칭한다.
- 외부에서 프로퍼티에 접근할 시 메서드(getter/setter)를 통해서만 가능하다.
등
#### 기업의 시스템 규모가 커짐에 따라 사용자 요구를 빠르고 안정적이게 처리하기 위한 다양한 기술(트랜잭션, 멀티 쓰레딩 등)이 요구되기 시작함 → 이를 해결하고자 등장 → EJB를 통해 개발자들이 도메인과 비즈니스 로직에만 집중할 수 있도록 함. 스프링 탄생 전 스프링과 JPA 등의 기술을 사용할 수  있는 자바의 표준 기술이었음.
- Enterprise Bean: 비즈니스 로직을 실행하는 서버 컴포넌트
    - 세션 빈, 엔티티 빈, 메시지 구동 빈
- Container: enterprise bean에 대한 런타임 환경 제공. EJB와 enterprise bean 간 통신을 해주는 역할
- EJB server: 컨테이너를 관리해서 EJB로서 필요한 시스템 서비스(데이터베이스 처리, 트랜잭션 처리 등)을 구현
- 특징
    1. 인스턴스 풀링: 객체를 미리 생성하고 메모리에 저장하여 사용 준비 상태에서 서버가 동작 → 동시 접속자에 대한 안정성 지원
    2. 트랜잭션 처리: 컨테이너가 자동으로 모든 메소드에 대한 트랜잭션을 처리 → 안정적인 데이터 조작
    3. 퍼시스턴스 관리: 빈즈의 상태를 메모리에서 사용여부에 따라 자동으로 활성화, 비활성화를 관리
    4. 분산 기능 지원
#### Serverlet: 동적 웹페이지를 생성할 때 사용하는 자바 기반의 웹 애플리케이션 컴포넌트(클래스). 간단한 메서드 호출만으로 다룰 수 있게 지원. 클라이언트의 요청을 처리하고 응답하는 작은 서버 측 프로그램.
- 자바의 스레드를 이용하여 동작
- MVC패턴에서 컨트롤러로 이용
- 컨테이너에서 실행
- Servlet Container: 구현되어 있는 서블릿 클래스의 규칙에 맞게 서블릿을 담고 관리해주는 컨테이너.
- **동작 과정(Servlet Container)**:
    1. 클라이언트 요청
    2. HttpServerletRequest, HttpServerletResponse 객체 생성
    3. 해당하는 서블릿에서 service() 메소드 호출
    4. doGet() 또는 doPost() 메소드 호출
    5. 동적 페이지 생성 후 ServerletResponse 객체에 응답 전송
    6. 2번의 두 객체 소멸
- 서블릿의 생명주기: 초기화→서비스 수행→소멸하기까지의 과정. 각 단계마다 호출되어 기능을 수행하는 콜백 메서드를 서블릿 생명주기 메서드라고 함.
    1. init(): 서블릿 요청 시 맨 처음 한 번만 호출
    2. doGet(), doPost(): 서블릿 요청 시 매번 호출(service 메소드를 통해), 실제로 클라이언트의 요청 작업을 수행
    3. dedstroy(): 서블릿이 기능을 수행하고 메모리에서 소멸될 때 호출

### 스프링 프레임워크는 어떻게 탄생하게 되었나?
#### 스프링 등장 이전 EJB가 애플리케이션의 업무 로직을 담당하는 자바 표준 기술이었지만, 클래스 간 의존도가 커지는 등 복잡성이 문제로 제기되기 시작하면서 이를 해결하기 위해 2가지의 오픈 소스가 등장.
1. 로드 존슨의 J2EE Design and Development: 이 시스템의 예제 코드들(30,000라인)이 지금의 스프링 기본 코드가 됨 → EJB 컨테이너 대체. 해당 예제 코드에는 현재 스프링의 핵심 개념과 기반 코드인 BeanFactory, ApplicationContext, POJO, *IoC(제어의 역전)*, *DI(의존성 주입)*등이 포함되어 있음.
2. 개빈 킹의 하이버네이트 ORM(Object Relation Mapping) 프레임워크: EJB 엔티티 빈 기술을 대체하고 JPA(Java Persistence API)의 새로운 표준 정의. JPA는 표준 인터페이스, 하이버네이트는 JPA의 구현체

### 스프링이란?
#### 엔터프라이즈급 애플리케이션을 개발하는 데 필요한 모든 기능을 제공하는 오픈 소스 프레임워크이자 생태계. <br> \*스프링 생태게: 스프링 프레임워크, 스프링 부트와 함께 스프링 데이터, 스프링 세션, 스프링 시큐리티, 스프링 RestDocs, 스프링 배치, 스프링 클라우드 등으로 확장
### 스프링의 구성요소는?
#### 주요 모듈
1. Core Container: IoC/DI, Bean 정의 및 관리 등 스프링의 핵심 기능 제공
2. Data Acess/Integration: JDBC, ORM 등 데이터 접근 기술을 일관성 있게 사용할 수 있도록 지원
3. Web: Spring Web MVC를 포함하여 웹 애플리케이션 개발에 필요한 기능 제공
4. AOP: 관점 지향 프로그래밍을 지원하는 모듈
### MVC 패턴이란? 패턴의 동작 방식은?
#### MVC(Model-View-Controller): 애플리케이션을 세 가지 역할로 분리하여 책임 분리를 달성하는 소프트웨어 디자인
1. 모델: 애플리케이션의 데이터를 관리하는 부분 또는 로직
2. 뷰: 데이터가 화면에 보이는 부분을 관리
3. 컨트롤러: 사용자의 입력을 처리하고, 모델과 뷰 사이에서 상호작용 할 수 있도록 지원 → (웹에 적용할 시) 사용자가 웹 페이지에서 어떤 동작을 취하면 이를 감지하고, 그에 따라 모델의 데이터를 변경하거나 뷰를 업데이트 함.
#### MVC 동작과정
1. 사용자의 요청이 컨트롤러로 들어옴
2. 컨트롤러가 해당 요청을 처리하기 위해 모델을 호출
3. 모델은 비즈니스 로직을 수행하고 데이터를 가공
4. 컨트롤러는 모델의 처리 결과를 바탕으로 뷰를 선택
5. 뷰는 모델에서 받은 데이터를 기반으로 화면을 구성하여 사용자에게 응답
### 스프링과 스프링부트의 주요 차이점?
- 스프링과 스프링부트는 모두 스프링 프레임워크를 기반으로 한 자바 웹 개발 프레임워크
1. 스프링은 포괄적인 프레임워크, 스프링 부트는 스프링 애플리케이션을 더 쉽게 설정하고 시작할 수 있도록 돕는 서브 프레임워크
2. 스프링은 개발자가 직접 스프링 컨테이너를 구성/빈 객체를 등속/빈 객체 간 의존성을 설정해야 함. 반면, 스프링 부트는 이러한 것들을 자동 처리

### 스프링 프레임워크의 주요 특징은?
1. 경량 컨테이너: EJB 컨테이너에 비해 훨씬 가볍고, POJO 기반으로 비즈니스 로직을 쉽게 구현할 수 있음.
2. Ioc, DI: 객체 생성, 관계 설정, 생명주기 관리를 개발자가 아닌 프레임워크가 대신함.
3. 관점 지향 프로그래밍(AOP):횡단 괌심사(보안, 로깅, 트랜잭션 등)를 핵심 비즈니스 로직과 분리하여 모듈화
4. 트랜잭션 관리의 추상화: 다양한 트랜잭션 기술(JDBC, JPA 등)을 일관된 방식으로 사용할 수 있도록 지원함.

### DI란?
#### IoC를 구현하는 구체적인 디자인 패턴
- 의존성: 한 객체가 다른 객체를 필요로 하는 관계
- 주입: 필요한 객체를 개발자가 직접 생성하거나 찾아서 연결하는 대신, 외부(스프링 컨테이너)에서 객체를 만들어서 사용하려는 객체에게 전달(주입)해주는 방식
- 구현 원리: 생성자, Setter 메서드, 또는 필드를 통해 이루어지며, 이를 통해 코드 변경 없이 객
체 간의 관계를 유연하게 바꿀 수 있음(결합도 낮춤).
### IoC란?
#### 객체의 생성, 생명주기 관리, 의존성 관계 설정 등의 제어권을 개발자 코드에서 프레임워크(스프링 컨테이너)로 넘기는 소프트웨어 디자인 원칙
- 역전된 제어: 전통적인 프로그래밍에서는 개발자가 new 키워드를 사용하여 직접 객체를 생성하고 관리했지만 IoC 환경에서는 스프링이 필요할 때 객체를 생성하고 제공해줌
- 개발자는 핵심 비즈니스 로직에만 집중하고, 복잡한 객체 관리는 스프링에게 맡길 수 있
어 코드의 유연성과 재사용성이 높아짐.
### AOP란?
#### AOP (Aspect-Oriented Programming, 관점 지향 프로그래밍)은 핵심적인 관점과 부가적인 관점으로 나누어서 보고, 그 관점을 각각 모듈화 하는 것. = 여러 군데 흩어져 있는 횡단괌심사를 하나의 모듈로 분리하여 재사용 할 수 있게 하는 것.
- 횡단 관심사: 핵심 기능을 가로지르는 기능. <br> Ex. 은행 시스템에서의 핵심 관심사: 입금/출금/이체, 횡단관심사: 보안 및 예외처리 등
- AOP는 이러한 횡단 관심사를 *애스펙트(Aspect)*라는 별도의 모듈로 분리하고, *위빙(Weaving)*이라는 과정을 통해 핵심 로직의 특정 시점(메서드 호출 전후 등)에 끼
워 넣음.
- 장점: 핵심 비즈니스 로직이 부가 기능들로 인해 오염되는 것을 막아 코드의 응집도를 높이 재사용성을 높.임

### Servlet Container란?
#### Servlet Container: 서블릿이 동작하는 공간으로 서블릿 객체를 생성, 초기화, 호출, 종료하는 생명주기를 관리할 뿐만 아니라, 클라이언트의 요청을 받고 응답할 수 있도록 웹 서버와의 소켓 통신 지원 + 멀티스레드 지웜 및 관리, 선언적 보안 관리

### Spring Container란?
#### 스프링 컨테이너는 스프링 프레임워크의 핵심 요소로, 자바 객체의 생명 주기를 관리하며, 생성된 자바 객체들에게 추가적인 기능을 제공함.

### DispatcherServlet이란?
#### Spring Web MVC에서 클라이언트의 요청을 가장 먼저 받는 프론트 컨트롤러 패턴의 구현체. <br> *프론트 컨트롤러:* MVC 구조에서 사용되는 디자인 패턴으로 서블릿 컨테이너의 맨 앞에서 서버로 들어오는 클라이언트의 모든 요청을 받아서 처리해주는 컨트롤러.
### 프론트 컨트롤러 패턴이란?
#### 웹 애플리케이션의 모든 요청을 하나의 중앙 집중식 핸들러(프론트 컨트롤러)가 처리하도록 하는 디자인 패턴
- 모든 요청에 대해 공통적으로 필요한 작업(인증, 로깅 등)을 단일 지점에서 처리하므로 코드 구조화 및 유지보수 간편
#### DispatcherServlet 이전에는 각각의 서블릿을 직접 만들고 해당 서블릿의 URL 매핑을 위해 web.xml에 등록해 주는 과정이 필요했지만 잘 구현된 DispatcherServlet이 애플리케이션으로 들어오는 모든 요청을 핸들링해 주고 공통 작업을 처리해 주면서 개발자가 각각의 서블릿을 만들고 등록하는 번거로움이 줄어들게 되었음.
### Application Context란?
#### 스프링 컨테이너의 인스턴스로, 어플리케이션의 빈(Bean) 객체들을 생성, 제어 관리함. <br> Spring에서는 빈의 생성과 관계설정 같은 제어를 담당하는 IoC(Inversion of Control) 컨테이너인 빈 팩토리(Bean Factory)가 있지만 빈의 생성과 관계설정 외에 추가적인 기능이 필요함. 이러한 이유로 Spring에서는 빈 팩토리를 상속받아 확장한 애플리케이션 컨텍스트(Application Context)를 주로 사용.

### Servlet Container에는 무엇이 탑재될까?
#### (추가 예정)
